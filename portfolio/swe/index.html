문제 정의 및 설계 가설

B2C 핀테크 플랫폼은 학습 곡선이 높아 신규·라이트 유저가 이탈하는 문제가 있었습니다. 이를 해결할 방법으로 "이미 검증된 자산배분 전략을 단순한 인터페이스로 제공"하는 구독형 상품을 기획했고, 전략 라인업의 핵심축으로 거시 체제 분산형 자산배분 전략을 배치했습니다. 기관에서 정립된 자산배분 프레임워크는 개인 투자자가 바로 운용하기 어려워, 실전 운용 가능한 형태로 재설계하는 것이 핵심 과제였습니다.

전략을 "해결할 기술 문제"가 아닌 검증 가능한 가설로 구조화하고, 각 가설에 검증 방법과 반증 조건을 함께 정의했습니다. 가설이 기각되면 시스템 전체가 아니라 해당 부분만 수정하도록 모듈을 설계 단계에서 분리했습니다 — 단위 테스트 실패 시 모듈 경계가 분명해야 디버깅이 빠른 것과 같은 원칙입니다. 또한 단일 시장 사이클에서 정적 비중 포트폴리오가 노출했던 구조적 취약성(2022년 주식↓ + 채권↓ 동시 하락 등)을 보완하는 것을 일차 목표로 두었습니다.

  • →H1. 시장 상태에 따라 자산 성과가 달라진다 — 자산의 기대수익은 무조건부 평균이 아니라 거시 체제 조건부 분포로 발현되므로, 정적 비중은 체제 전환기에 구조적으로 취약. 거시 신호를 엔진의 입력으로 직접 반영. 반증 조건: 단일 체제 가중 포트폴리오가 일관되게 초과 수익을 내면 체제 분산의 가치는 낮음
  • →H2. 자본 비중 균등은 리스크 균등이 아니다 — 자본 비중이 균등(1/N)이어도 변동성·상관구조에 따라 자산별 리스크 기여도는 극단적으로 불균등 (60/40 포트폴리오에서 주식 리스크 기여도가 90%를 상회). 따라서 각 자산의 리스크 기여도가 사전 지정된 비율에 비례하도록 배분. 반증 조건: 리스크 균등 포트폴리오가 자본 균등 대비 risk-adjusted 성과 우위를 보이지 않으면 가치 노이즈
  • →H3. 최적화 결과의 품질은 입력 품질에 종속된다 — 자산 수가 많고 관측 기간이 제한된 환경(N≈20)에서는 입력 행렬이 수치적으로 폭주해 최적화 결과를 불안정하게 만듦. 입력 추정 품질이 최적화 알고리즘의 정교함보다 결과에 미치는 영향이 큼. 입력 전처리를 별도 설계 대상으로 분리. 반증 조건: 여러 추정 방식의 입력을 바꿔 넣어도 최종 비중 차이가 거래비용 수준 이하면 입력 품질의 영향은 제한적
  • →H4. 상시 헤지는 비용 측면에서 비효율 — 전(全) 기간 헤지는 비용 누적으로 기대수익률이 음수에 가까움. 꼬리 위험 임계 초과 시에만 노출도를 동적으로 축소하는 조건부 디레버리징이 장기 누적 수익을 보존. 반증 조건: 정적 80% 노출이 동적 축소와 위험-수익 모두에서 동등하면 dynamic의 부가가치 없음
  • →H5. 단일 추정의 극단성은 전략적 사전과의 결합으로 완화 — 단일 데이터 추정에 의한 극단 비중을 완화하기 위해 데이터 기반 최적해와 전략적 사전(prior)을 볼록결합. 데이터로만 풀면 OoS에서 불안정하다는 결과를 존중하면서, 꼬리·체제 전환에 대한 추정 오차를 prior anchor가 완충. 반증 조건: prior 가중이 0일 때가 모든 지표에서 우세하면 prior는 노이즈

구조적으로는 전략적 사전(Investor Views) + 데이터(공분산) + 상황 신호(레짐)를 결합해 하나의 비중 벡터로 수렴시키는 구조입니다. 세 가지 성격이 다른 정보원을 한 엔진 안에서 조합하는 것이 핵심이며, 각 정보원의 가중치는 상품 레벨별 파라미터로 조정됩니다.

시스템 아키텍처 — 모듈 구성 및 데이터 흐름
주 파이프라인: 데이터 → 레짐·꼬리위험 피처 → 포트폴리오 최적화 → 리스크 오버레이 → 실전 집행 · 레짐 감지는 자산 가중치 입력에, 꼬리 위험은 오버레이 트리거에 반영 · 통계적 선행성 검증은 신규 피처 도입 시 게이트로 별도 운용
핵심 설계 판단
최적화 입력 품질 — 4단계 전처리 파이프라인
기존 단일 단계 최적화 방식은 입력 데이터의 추정 오류에 취약해 실전에서 소수 자산에 비중이 쏠리는 문제가 있었고, 단순 균등 분산은 자산군 간 균형과 자산군 내부 효율성을 동시에 잡지 못했습니다. 이 두 문제를 분리해서 푸는 방향으로, 자산군 간 리스크 균형을 먼저 맞추고 자산군 내부에서 효율성을 조절하는 2단계 구조로 재설계했습니다.

입력 품질이 최적화 결과의 핵심 결정 요소라, 단일 연산이 아닌 4단계 전처리 파이프라인으로 구성했습니다. 각 단계는 이전 단계의 출력을 전제로 다음이 동작하는 구조이며, 한 단계라도 생략하면 이후 가정이 깨집니다.
  • →1단계 — 시계열 가중 평균: 최근 관측에 더 큰 가중치를 부여해 시장 상태 변화에 대한 반응성 확보. 단순 평균은 체제 전환에 반응이 느림
  • →2단계 — Shrinkage 보정: 자산 수 대비 관측 기간이 부족해 입력 행렬이 수치적으로 불안정해지는 문제를, 안정적인 기준 행렬 쪽으로 보정해 완화. 이 단계 없이는 이후 솔버가 자주 실패
  • →3단계 — 노이즈 제거: shrinkage가 대체하지 못하는 잔여 노이즈를 무작위 데이터 수준의 신호는 노이즈로 간주해 제거. 최종 비중이 노이즈에 덜 흔들리게 함
  • →4단계 — 수학적 조건 보정: 솔버가 요구하는 수학적 조건을 만족시키는 최종 가드. 극히 작은 수치 오차로 솔버가 입력을 거부하는 경우를 방지
임의 조합이 아니라 각 단계가 서로 다른 문제를 해결하며, 어느 것도 다른 것으로 대체되지 않는다는 것이 설계의 핵심입니다.
시장 상태 감지 — 해석 가능성 우선의 규칙 기반 설계
실험 단계에서 비지도 학습 기반 클러스터링(데이터로부터 자동으로 시장 상태를 군집화하는 ML 방식, HMM·GMM 등)도 구현해봤지만, 출력 클러스터에 경제적 의미를 사후적으로 붙여야 한다는 근본 한계가 실서비스 신뢰성에서 문제가 됐습니다. 대신 경제적 의미(성장·인플레 축)가 설계 단계에서 직접 반영되는 규칙 기반 레짐 감지를 프로덕션에 올려, 해석 가능성과 운영 안정성을 우선하는 선택을 했습니다.
꼬리 위험 관리 — 복수 지표 통합
단일 지표에 의존하면 극단 국면에서 신호를 놓치는 경우가 많다는 교훈이 바탕입니다. 특히 단순히 과거 분포만 보고 추정하는 방식은 꼬리(극단 손실 구간)를 과소추정하는 한계가 있어, 성격이 다른 세 가지 지표를 결합하는 방향으로 설계했습니다. 변동성 동적 추정은 현재 시장의 불안정성을, 극단값 분포 모델링은 꼬리의 분포 자체를, 이상 수익률 탐지는 과거 패턴에서 벗어난 움직임을 각각 포착합니다. 판단 오류의 상관관계가 낮은 세 신호를 통합해 한 지표가 놓친 신호를 다른 지표가 포착하도록 하는 구조이며, 2022 고변동성 국면 전례가 이 다층 방어 설계의 직접적 동기였습니다.
자산 유니버스 정제 — 운용 데이터 기반 v2 진화
v1의 운용 데이터와 자산 간 상관관계를 분석해 경험적 검증에 기반한 유니버스 정제를 v2에서 단행했습니다. 핵심 변경은 다음과 같습니다.
  • →중복 노출 자산 제거 — 다른 보유 자산과 rolling correlation 0.91~0.99로 독립적인 분산 효과를 제공하지 못하는 종목은 유니버스에서 배제
  • →중복 듀레이션 자산 제거 — 다른 보유 자산과 역할이 중복되며 양방향 체제에서 모두 비효율을 보였던 종목 배제
  • →장기 듀레이션 자산 복원 — 디플레이션 국면 방어를 위한 전용 자산을 명시적으로 편입해 체제 분산의 한 축 회복
  • →비상관 자산군 추가 — 다른 자산군과 약한 상관관계를 가지면서 위기 시 비대칭 수익을 제공하는 카테고리 신규 편입
v2 최종 유니버스는 8개 자산군 카테고리로 정제되었으며, 각 카테고리는 (i) 특정 거시 사분면에 대한 조건부 우위 + (ii) 독립적인 위험 노출이라는 두 제약을 통과하도록 설계.
리스크 배분 비율 — 운용 환경 제약을 반영한 보정
기관 표준의 4분면 균등 분산(25/25/25/25)은 1.6×~1.8× 레버리지를 전제로 하므로 일반 투자자 환경에서는 산술적으로 실현 불가합니다. 무차입 환경에서 동일한 균형을 강제하면 수학적으로 풀리지 않는 제약이 발생하기 때문에, 거시 사분면별 발생 빈도 × 조건부 우위를 반영한 비대칭 리스크 배분(대략 56:22:16:6 비율)으로 재설계했습니다. 설계 의도는 in-sample 최적이 아닌 out-of-sample 안정성 우선 — 단일 추정에 기반한 in-sample 최적해는 미래 데이터에서 불안정하므로, 보수적 배분이 OoS 안정성 프리미엄을 가져온다는 위 H5 가설(prior anchor)의 직접 구현입니다.
자산군 분리의 정량 정당화 — 분산 효과 측정
섹터 카테고리와 일반 주식 카테고리의 상관관계 ρ ≈ 0.85가 외부 비판으로 가장 자주 제기됩니다 — "사실상 중복 노출 아닌가?". 이에 대한 정량 답:
  • →분산 효과 정량 측정 — 정규화 리스크 분포에서 effective bet count ≈ 6.5 (8 nominal class 중). 엔트로피 기반 분산 손실은 약 19%만 발생
  • →통합 시 기회비용 — 섹터를 일반 주식으로 합치면 effective bet count ≈ 5.2로 추가 20% 손실
  • →자산군 내부 차별화 경로 보존 — 섹터 종목 간 효율성 차이가 자산군 내부 배분 단계에서 직접 표현되는 경로 보존이 분리의 일차 목적. 통합 시 테마별 신호(AI · healthcare 등) 산출 경로가 닫힘
결론: 섹터 분리는 분산 효과의 추가 이득(19%)보다 자산군 내부 차별화 산출 경로 보존이 일차 목적이며, 8-class 정의에 명시적으로 인코딩된 설계 의도입니다.
백테스팅 인프라 · 성과 검증

시뮬레이션 결과로 라이브 운용 의사결정을 내리는 시스템에서는 시뮬레이터 자체의 정확성이 의사결정 품질을 결정합니다. 시뮬레이션 환경을 별도 설계 대상으로 다뤘고, 라이브와 시뮬레이션의 재현성 확보, look-ahead bias 구조적 차단, 시계열 분할의 이론적 정당화, 가설 기반 반증 구조, 거래비용·시그널 lag 강건성 검증이 핵심 축입니다.

① 시계열 분할 — Walk-Forward Optimization (WFO)

  • →이론적 정당화: 단일 train/test 분할은 분할 시점 선택 편향과 반복 평가에 따른 in-sample fit 변질에 노출되므로, multi-fold OoS 평균을 objective로 채택해 단일 fold의 우연 fit이 평균에서 희석되도록 강제 (Pardo 2008 The Evaluation and Optimization of Trading Strategies의 walk-forward 표준)
  • →WFO 표준 구현 — 라인업 일관 검증 인프라: 라인업 중 hyperparameter 자유도가 큰 전략에 동일 WFO 표준을 적용. 표준 구현 구성:
    • ·n_folds = 4 (rolling window) · train_days = 252 (≈1 year) · test_days = 63 (≈3 months)
    • ·gap_days = 5 (Embargo) — train 종료와 test 시작 사이의 buffer로, 시계열 자기상관에 의한 train-test 정보 leakage를 차단. López de Prado (2018) Advances in Financial Machine Learning Ch.7의 Embargoed Cross-Validation 표준 적용
    • ·Objective: Score(θ) = (1/|F|) Σ Sharpe^OoS_f(θ). Bayesian optimization 기반 sampler + 부진 trial 조기 종료 결합으로 search 효율 확보
  • →본 자산배분 시스템의 WFO 미적용 — 정당화: 본 시스템은 튜닝 자유도가 작도록 설계했고(상품 레벨별로 변동되는 free hyperparameter는 소수), 그보다 구조적 design 위험(자산 유니버스 구성·자산군 분리·시장 상태 정의)이 핵심 검증 대상이라 판단. 데이터 측면 한계도 함께 작용 — 백테스트 기간 4.32년은 multi-fold 평가의 통계적 한계 효용을 줄임. 대신 시장 상태 분류기의 point-in-time 사전 검증(2017–2024, 13개 test dates에서 핵심 축 100% / 통합 분류 77% 정확도) + Leave-one-out + effective bet count 분석을 적용해, *parameter optimization 위험*이 아닌 structural design 위험에 검증 자원을 집중

② Look-ahead bias 차단 — 4 메커니즘

  • →Point-in-Time (PIT) 데이터 처리 — 외부 신호(매크로 지표, 거시 데이터)는 release date 기준으로 indexing해 시뮬레이터 timeline에서 *발표 시점부터 사용 가능*하도록 lag 적용. as-of가 아닌 available-at 기준이 시계열 시뮬레이션의 표준
  • →신호 생성 함수 구조적 동형성 — 동일 시그널 생성 함수가 라이브/시뮬레이션에서 동형으로 호출. 라이브는 DB의 직전 승인 비중, 시뮬레이션은 시뮬레이터의 직전 승인 비중을 참조. 함수 시그니처와 입력 데이터 구조 일치로 코드 변경을 최소화하는 구조를 채택해, *"시뮬레이션과 라이브에서 다르게 동작"*의 가장 흔한 원인인 *상이한 데이터 흐름*을 구조적으로 차단
  • →WFO Embargo 메커니즘 (라인업 일부 전략에 한정 적용) — 위 ①의 gap_days=5는 시그널 자기상관에 대한 buffer로 작동. 시계열 데이터는 단기 자기상관(Lo & MacKinlay 1988)을 가지므로, train 종료 직후 test 시작 시 동일 정보가 train-test 양쪽에 포함되어 OoS가 인위적으로 부풀어짐. 5일 embargo는 일별 시그널의 1주 horizon을 cover
  • →시그널 상태 3-tier 분리 — "최적화는 매일, 리밸런싱은 임계 통과 후, 체결은 또 다른 상태"의 세 가중치 도메인을 명시적으로 분리해 시뮬레이션·라이브에서 동일한 의미를 갖도록 강제. 3개 도메인의 혼동이 *가장 흔한 look-ahead leakage 패턴* — 최신 최적화 산출물 / 직전 승인 비중 / 실 포지션 상태를 별도 저장 슬롯으로 관리

③ 거래비용·슬리피지 — 현재 처리와 한계 declaration

  • →현재 상태 — gross 기준 시뮬레이션: 본 시뮬레이션은 gross return 기준이며 거래비용·슬리피지·시장충격을 명시 모델로 차감하지 않음. 자산배분 전략의 turnover가 본질적으로 낮아 net-gross 괴리가 작다는 *경험적 가정*에 기반하지만, 공식적인 net 성과 검증은 후속 과제로 정직하게 분리. 학술 표준(Frazzini, Israel & Moskowitz 2018 Trading Costs of Asset Pricing Anomalies) 기준 자산배분류 전략의 거래비용은 risk-adjusted 성과를 약 10-20bps 후퇴시키는 수준
  • →완화 메커니즘 — 임계 기반 deadband: 시뮬레이션에서 직접 net 비용을 모델링하는 대신, 리밸런싱 임계(deadband)를 채택해 turnover 자체를 구조적으로 낮춤. 매일 최적화는 수행하되 *직전 승인 비중 대비 최대 비중 편차가 임계값 이하면 리밸런싱 skip*. 일별 noise가 turnover로 변환되지 않도록 차단해, 자산배분 표준 대비 낮은 turnover 영역에 위치
  • →거래비용·시장충격 스트레스 테스트 — 후속 과제: Phase 2에서 정식 도입 예정.
    • ·거래비용 stress: 다단계 거래비용 시나리오에서 net Sharpe·CAGR 변화 측정. 어느 cost level에서 BM 대비 우위가 소멸하는가를 정량 답
    • ·시장충격 stress: 자산별 비중이 일별 거래량 대비 임계 초과 시 시장충격 적용. 본 시스템은 자산당 비중 범위가 단일 사용자 운용에서는 무관하나 AUM 확장 시 critical. Almgren & Chriss (2000) implementation shortfall 모델 적용 검토

④ 시그널 Lagging 강건성 — 발생 원천과 대응

  • →Lag의 두 발생 원천:
    • ·Data lag — 매크로 지표의 release lag (CPI ~2주, GDP ~4주). 시그널 입력 자체가 과거 정보 기반
    • ·Execution lag — 시그널 생성 시점부터 실제 체결까지의 시차 (본 시스템은 일별 평가 + 다음 거래일 진입)
  • →현재 처리 (구현 완료): 시뮬레이터는 다음 거래일 진입 원칙으로 동작 — 시그널이 t 시점에 생성되면 t+1 시점에 체결. 동일 bar 내 진입(look-ahead의 한 형태)은 구조적으로 차단됨. 매크로 신호의 release lag는 위 ②-PIT 처리로 cover
  • →Lag 스트레스 테스트 — 후속 과제: Phase 2에서 정식 도입 예정.
    • ·추가 1-2일 lag 시나리오 — 시그널 생성 후 추가 1일 / 2일 지연 후 진입했을 때의 성과 변화. *운영 사고(배치 실패, 데이터 지연) 시 alpha persistence* 측정
    • ·시그널 frequency degradation — 일별 평가 → 주별 평가로 frequency를 낮췄을 때 alpha persistence. 시그널이 빠르게 decay하는지를 측정해 운영 안정성 위협 판단

⑤ 가설·구조 검증 — 결과 편향 차단

  • →가설 기반 반증 조건 사전 정의 — 각 가설마다 성과 기준이 아니라 반증 조건을 설계 단계에서 정의. 시뮬레이션 결과가 좋게 나와도 반증 조건이 충족되면 구조적 재설계 근거로 활용. "좋은 성과를 설명하는 이유"가 아니라 "가설이 언제 기각되는가"를 먼저 합의해 결과 편향을 억제
  • →Leave-one-out 종목 기여도 검증 — 자산 유니버스 정제의 정당성을 사후 검증하기 위해 각 종목을 1개씩 제외했을 때 portfolio Sharpe 변화를 측정. 2022–2025 시뮬레이션 기간에서 19개 종목 모두 positive marginal Sharpe contribution (가장 큰 contribution은 특정 자산 제외 시 ΔSharpe -0.21, 다른 자산들은 -0.06~-0.08 순)을 확인. 이는 v2 유니버스의 추가 reduction이 시뮬레이션 범위에서 정당화되지 않음을 의미하며, 자산 유니버스 정제 결정의 boundary를 데이터로 입증
  • →자산군 분리 민감도 — Effective Number of Bets — 두 자산군 카테고리 간 상관 ρ ≈ 0.85가 외부 비판으로 가장 자주 제기되는 부분. 이에 대해 effective bet count 메트릭으로 정량 답: v2의 정규화 리스크 분포에서 effective bet count ≈ 6.5 (8 nominal class 중). 두 카테고리를 통합하면 ≈ 5.2로 추가 손실 — 분리 유지가 분산 효과 측면에서 정당

⑥ 시뮬레이션 ↔ 라이브 정합성 + 시스템 경계

  • →시뮬레이션 ↔ 실 운용 일관성 검증 — 핵심 OoS(out-of-sample) 성능 지표는 실 운용에서 시뮬레이션의 risk-adjusted 우위가 유지되는가. v2 시뮬레이션 Sharpe 1.15 ↔ 실운용 통합 Sharpe 1.18 (+0.03)로 정합 일관 확인, drawdown 압축도 시뮬레이션 3.2× / 실운용 1.8× 동일 방향 작동. 실운용 표본 1.36년 한계는 명시하되 방향성과 부호 일치는 robust signal로 1차 평가
  • →집행 위임 — 전략·집행 분리 원칙 — 최종 주문 집행은 Project 02의 분산 트레이딩 · 주문 집행 엔진에 위임되며, 전략 로직과 집행 인프라가 수평 확장 가능한 형태로 분리. 시뮬레이션에서 검증된 시그널이 라이브에서 동일하게 재현되는 것은 이 위임 경계가 명확하기 때문
운용 성과 — 시뮬레이션과 라이브의 일관성 검증

동일 엔진을 v1(7 자산군 24종 / weekly 운용) → v2(8 자산군 19종 / 자산 유니버스 정제 + 신규 자산군 편입 / daily 운용)로 진화시켰습니다. 시뮬레이션과 라이브 운용 모두 v1·v2를 분리해 측정한 뒤, 통합 운용 결과를 함께 제시합니다. 핵심 메시지는 "백테스트의 risk-adjusted 우위가 실 운용에서도 유지되는가"이며, 이것이 OoS(out-of-sample) 일반화 검증의 정의입니다.

운영 파라미터 관리 및 모델 고도화

퀀트 전략은 한 번 만들고 끝나는 것이 아니라, 시장 환경 변화에 따라 리스크 파라미터와 모델을 지속적으로 재조정해야 합니다. 이 전략에서 조정 대상이 되는 파라미터 축과 재조정 루틴은 다음과 같습니다.

  • →조정 대상 리스크 파라미터 — ① 리밸런싱 임계값(최대 비중 편차) — turnover vs tracking error의 trade-off를 결정 ② 꼬리 위험 오버레이 임계 — 변동성·극단값 지표의 발동 기준 ③ 레짐 감지 확률 임계 — 체제 전이 판정 민감도 ④ L1~L5 상품별 공격성 파라미터 3축(Strategic Weights 가중·Sharpe 틸트 강도·레짐 민감도). 각 파라미터는 데이터 기반 민감도 측정 이전이라도 시장 국면별 behavior가 명시되어 있어 변경 시 영향 범위를 사전에 예측 가능
  • →정기 모니터링 — 주 단위 — 지표별 실전 성과를 벤치마크와 대비해 Sharpe·MDD·자산군 기여도 추이를 추적. 일정 수준 이상의 성과 저하가 지속되면 원인 분석 우선순위에 올려 파라미터 재조정 또는 모델 고도화 대상으로 분류
  • →피처 유효성 재검증 — 반기 단위 — 포트폴리오 최적화에 사용 중인 피처도 과거 한 번 검증했다고 영구 유효하다고 가정하지 않음. 하위 사례에서 다룬 통계적 선행성 검증(인과성·정보 전달량 측정)을 반기 단위로 재실행해 여전히 유의한지 확인하고, 기각 시 피처를 재설계 또는 대체

금융 도메인 신뢰성 설계 포인트

주문 체결은 실패 시 실제 자산 손실로 직결되는 도메인이라, 메시지 레벨의 유실 방지 외에도 주문 단위·계좌 단위·운영 단위 각 층위에서 추가 안전장치를 설계했습니다.

  • →TWAP 과체결 방지 3중 방어 — 라운드 시작 시 ① 미체결 주문 취소 → ② 보유 수량 재조회로 실 상태 재계산 → ③ 최종 목표 수량으로 주문 규모 클램핑. 장중 포지션이 흔들려도 누적 주문이 목표를 초과하지 않음
  • →장 마감 기준 라운드 매 틱 재산정 — 총 주문 시간/주문 간격으로 기본 라운드를 산정하되, 장 마감까지 남은 시간을 기준으로 매 틱 재계산해 장 종료 이후 주문이 발생하지 않도록 차단
  • →오류 코드 분류 기반 재시도 — 거래소 오류를 무시 가능 / 재시도 가능으로 분류해 라운드 루프의 안정성 확보. 포괄적 예외 처리로 인한 "실패 원인 불명" 상황 차단
  • →계좌 자동 일시중단 (Circuit Breaker · Bulkhead 효과) — 특정 계좌에서 거래소 오류가 반복되면 해당 계좌를 자동 비활성화(Circuit Breaker open). 한 계좌의 문제가 전체 집행 루프로 연쇄 확산되는 것을 격벽으로 차단하는 Bulkhead 효과를 부수적으로 확보
  • →시간 슬롯 단위 시그널 중복 방지 — at-least-once 메시징의 필연적 중복 수신을, 시간 슬롯 기반 캐시 키로 동일 슬롯의 재처리를 차단. 금융 도메인에서 중복 주문은 곧 손실이므로 이 층위가 필수
  • →중앙 예외 파이프라인 — 예외 로깅·Slack 알림·DB 기록을 단일 지점으로 통합. 개별 태스크에 흩어진 에러 처리 대신 관찰 가능성과 원인 분석을 일원화
  • →ML 서빙 품질 자동 감시 — 학습→예측→재학습 생애주기 관리와 3계층 검증 체계(합성 데이터 유닛 테스트 · 실데이터 기반 통합 검증 · 정량 보고서)로 모델 품질을 자동 모니터링. 모델 오염이 실 주문에 영향을 주지 않도록 사전 차단
성과
1,600+ 계좌 월평균 거래대금 약 5천억 24/7 무중단 운영 6개 거래소/증권사 통합 메시지 유실 0건

Binance · Bitget · OKX 선물 + Upbit 현물 + KIS · DB금융투자 주식을 단일 아키텍처로 통합 운용 중입니다. 시그널 생성부·주문 집행부 분리로 부분 장애가 전체를 중단시키지 않으며, 수평 확장이 계좌 단위·거래소 단위로 가능해졌습니다. "메시지 유실 0건"은 publisher confirms로 broker ACK 검증 + manual ACK consumer + at-least-once + 시간 슬롯 멱등성으로 보장하는 형태이며 (exactly-once의 강한 주장은 아닙니다), DLQ에 적재된 메시지는 운영 중 모두 재처리해 손실 없이 회수해온 결과입니다.

이 설계가 일반화되는 도메인
1:N 메시지 브로드캐스트, at-least-once + 시간 슬롯 멱등성, DLQ 격리, Circuit Breaker 기반 격리는 거래 시스템에 한정된 기법이 아닙니다. 채팅·알림·실시간 동기화처럼 메시지를 다수 대상에 안정적으로 전달해야 하는 모든 시스템에서 동일한 신뢰성 요구를 만납니다. 이 시스템에서의 설계 판단은 도메인이 바뀌어도 그대로 옮겨갈 수 있도록 트레이드오프 형태로 분리해 두었으며, 위 메시지 브로커 대안 비교 표와 신뢰성 설계 포인트가 그 형태입니다.
현재 남은 한계와 부채

현재 시스템의 주요 한계는 세 가지이며, 각각 해결 조건과 재평가 시점을 함께 관리하고 있습니다. 첫째, at-least-once 메시징 기반이라 이벤트 재생·과거 시점 재현이 불가능해, 사후 분석이나 감사(audit) 요구가 커지면 별도 이벤트 스토어 도입이 필요합니다(대안 비교 테이블의 Kafka 전환 트리거와 연결). 둘째, 스케줄러·브로커의 HA 구성은 현 규모의 비용 효율이 맞지 않아 도입하지 않았고, 월평균 거래대금이 현재의 5배 이상으로 확장되거나 다운타임 허용치가 강해질 때 재평가 대상입니다. 셋째, TWAP 파라미터(주문 시간·간격)는 계좌별 설정은 가능하나 시장 상황에 따른 동적 조정 로직이 없어, 고변동성 국면에서의 체결 품질을 실측 기반으로 조정하는 로직이 중기 과제입니다.

탈락안 1 · Go — 인터페이스 계약의 명시적 표현력에서 차이
4개 거래소 어댑터의 공통 계약을 단일 인터페이스로 통일해야 하는 요구가 핵심이었습니다. Go의 structural typing(암묵적 인터페이스 충족)은 유연성 측면에서 장점이지만, 거래소가 N개로 확장될 때 새 어댑터가 계약을 부분 충족할 경우 실패가 다른 호출 지점에서 표면화되어 디버깅 표면적이 넓어집니다. Rust trait은 impl Trait for T를 명시적으로 강제하고 trait bound로 함수 시그니처에 제약이 박혀, 거래소 추가 시 "무엇을 구현해야 하는지"가 컴파일러에 의해 정확히 안내됩니다. 제네릭 성숙도(1.18+) 도 trait bound 수준의 제약 표현에는 미흡했습니다.
탈락안 2 · Java/Kotlin — JVM 런타임 비용
JVM 콜드 스타트(~2-5초) + JRE 포함 Docker 이미지 비대(~300MB+) 가 본 마이그레이션의 핵심 목표인 "메모리 효율로 계좌 확장 여력 확보"와 정면 충돌했습니다. Celery worker × 36의 메모리 절감 효과가 제한적입니다.
탈락안 3 · C++ — 메모리 안전성 부재
금융 시스템에서 use-after-free·buffer overflow는 자산 손실로 직결됩니다. 이 도메인에서 메모리 안전성을 컴파일러가 보장하지 못한다면 도입 자체가 위험을 늘리는 결정입니다. 비동기 생태계(Boost.Asio) 성숙도도 tokio 대비 부족했습니다.
선택 · Rust — 위 한계의 동시 충족
① GIL 없는 진정한 병렬성 (tokio M:N 스케줄링·JoinSet으로 1,000+ 계좌 전체 동시 spawn) · ② Zero-cost abstraction (제네릭 + trait bound·match 정적 디스패치 — 추상화에 런타임 비용 0) · ③ 컴파일 타임 안전성 (소유권 시스템 + 정적 타입 → 드라이버 호환성·**kwargs 시그니처 누락 같은 장애를 컴파일 시점에 차단) · ④ GC 부재 + 단일 바이너리 (Celery worker × 36 대비 메모리 절감) · ⑤ async 생태계 성숙도 (tokio + axum + sqlx + lapin + reqwest — 모두 프로덕션 검증 완료).
마이그레이션 전략 — Strangler Fig + Feature Flag

1,000+ 계좌가 실시간 운용 중인 시스템이라 단 한 번의 배포 실패도 허용되지 않는 전제에서 출발했습니다. Big Bang 전환을 배제하고, Python 서비스와 Rust 서비스가 RabbitMQ를 경계로 공존하며 환경변수 하나로 즉시 전환·롤백되는 구조를 채택했습니다.

참고로 Python 생태계 내 대안(arq·taskiq 등 async-native 태스크 큐)으로 Celery 한계를 완화하는 경로도 검토했으나, ① 운영 중 시스템의 마이그레이션 비용이 동등 시간의 Rust 전환 비용에 가깝고 ② 언어 수준 전환은 위 한계 패턴의 재발 자체가 구조적으로 불가능해진다는 점에서 영구 해결 효과가 더 컸기 때문에 후자를 선택했습니다.

3-mode Feature Flag · 동시 소비 차단
단일 환경변수로 운영 엔진을 전환합니다. Python 모드에서는 Rust executor가 즉시 return, Rust 모드에서는 Python consumer가 즉시 return하여 동시 소비로 인한 중복 주문을 코드 레벨에서 구조적으로 차단합니다. Shadow 모드에서는 Rust가 주문 수량만 계산하고 실주문은 전송하지 않아 실주문 없이 양 엔진 결과를 안전하게 비교할 수 있도록 설계했습니다. 전환 단위는 환경변수이므로 L1 롤백(Feature Flag)은 5초 이내이며, L2(이미지 롤백 ~20초)·L3(히스토리 복원)의 3계층 롤백으로 운영합니다.

검증된 효과 vs 미측정 항목 — 솔직한 한계
마이그레이션의 효과는 ① 자원 효율(메모리·이미지·LOC 실측) ② 특정 장애 클래스의 구조적 소멸(아래 표) 에서 확실히 검증되었습니다. 다만 지연(latency)의 before/after 정량 비교는 미수집입니다 — 전환 시점에서 "Shadow 파이프라인 완성 대기" vs "조기 전환 + 실거래 관찰" 의 트레이드오프를 따져, Python 시스템에서 동일 패턴 장애가 지속 발생 중이었고 Feature Flag 5초 롤백이 가능했기에 의식적으로 후자를 선택했습니다. 그 결과 전환 직후 hotfix 14건(1차 10건은 Shadow 사전 검출, 2차 4건은 실거래 관찰로 수정)이 발생했고, 이 비용은 사전에 예상한 범위 내였습니다. 이 경험은 "Shadow 사전 검출 + 실거래 관찰의 병용이 최소 위험 경로"라는 교훈으로 정리했고, 동등 상황 재현 시 Shadow 완성 우선 검토 가능성을 함께 기록해 후속 과제로 관리 중입니다.
학습 곡선과 적용 — 신규 언어 도입 방법론
Rust는 본 마이그레이션 시점에 본격적으로 처음 다룬 언어였습니다. 시스템 엔지니어링·분산 시스템·금융 도메인은 깊은 자산이었으므로, 학습 대상은 언어 고유 영역(소유권·라이프타임·async/Send/Sync·trait object·Pin/Future)에 집중되었고, 풀어야 할 문제 자체는 이미 익숙한 영역이었습니다. 운영 중 시스템에 직접 적용하기 위해 학습 → 격리된 설계(Strangler Fig + Feature Flag) → Shadow Mode 검증 경로 → 점진 전환의 시간 순서를 강제했고, 위험을 "언어 미숙"이 아니라 "전환 전략"으로 분리했습니다. 새로운 도구의 학습이 곧 운영 적용으로 이어진 사례이며, 채택할 도구가 변할수록 가치가 커지는 형태의 역량으로 본인의 강점을 자리매김하고 있습니다.

S3/Blob은 기본 역할이 파이프라인 산출물의 백업 저장소이지만, 동시에 여러 서비스 서버가 단일 소스에서 동일한 데이터를 읽어갈 수 있는 공용 조회 지점으로도 활용합니다. 각 서비스가 별도 데이터 복제본을 가지면 동기화 비용과 버전 드리프트가 발생하는데, 백업용으로 어차피 유지해야 하는 저장소를 읽기 경로로도 겸용함으로써 인프라를 늘리지 않고 "하나의 진실 소스(single source of truth)"를 확보하는 선택이었습니다. 파일 전송 기반 소비자(liveback·workstation)는 레거시 호환 목적으로 유지 중이며, 중기적으로 직접 조회 구조로 흡수할 예정입니다.

StoragePath 레지스트리는 price.adjusted, index.base, factor.eodhd 등 13개 논리 경로를 표준화한 추상화 계층입니다. 경로를 논리 이름으로 분리해두면 버킷 이관·환경 분리 시 코드 변경 없이 레지스트리만 갱신하면 되고, 위 결정 박스에서 다룬 Dual-Write 3단계 전환(기록→읽기 전환→단일화)도 이 추상화를 전제로 가능했습니다. MySQL은 메타데이터 전용으로만 두어(지원 종목 · 제외 티커 · 에러 로그) 시장 데이터 자체는 파일로 관리해 OLTP와 분석 워크로드의 부하를 분리했습니다.

품질 게이트 발동 시나리오: 팩터 생성 단계에서 신규 데이터가 직전 대비 5% 이상 감소하면 업로드를 자동 중단하고 백업 파일을 만들어 Slack으로 경고합니다. 실서비스가 잘못된 데이터로 오염되는 것을 사전에 차단하는 최후의 방어선 역할이며, EODHD 제외 티커는 주간 자동 리포트로 별도 추적됩니다.

현재 구조는 여러 소비자에게 동일 파일을 제공하는 "데이터 공급자" 역할에 최적화되어 있습니다. 소비자 수가 늘고 ML 기반 전략이 확장될수록 재사용성·거버넌스·학습-추론 일관성이 새로운 병목으로 부각될 것으로 보고 있으며, 두 축의 상용 수준 고도화를 단계적으로 진행할 예정입니다.
